# SKILL: Adding i18n Messages via lingo

## Purpose

This skill guides a coding agent in adding new i18n messages to the
`li18ngo`/`jaywalk` family of projects. The only hand-authored artefact is
an entry in the `Underliers` map. Lingo generates all Go code from that
entry - the generated output is never written or validated by hand.

---

## 1. The Golden Rule

The `*-auto.go` files in the `locale` package are generated by lingo and must
never be edited directly. To add, rename, or remove a message:

1. Edit `underlying-templ-data.go` only.
2. Run lingo.
3. Commit both the source change and the regenerated files.

---

## 2. Workflow - Adding a New Message

1. Add a new constant to the `UnderlyingType` enum in `underlying-templ-data.go`,
   following the existing iota pattern.
2. Add a corresponding entry to the `Underliers` map (see section 3).
3. Run lingo from the repo root:
   ```
   go run ./cmd/lingo/...
   ```
4. Verify the relevant `*-auto.go` file was updated.
5. Run tests to confirm nothing regressed.

> **SYNC NOTICE** - the `UnderlyingType` enum is duplicated in
> `cmd/lingo/main.go` to break the cyclic import between the generator and
> the locale package. When adding a constant to the canonical enum, follow
> the four-step checklist in the SYNC NOTICE comment at the top of `main.go`
> before running lingo.

---

## 3. `Underliers` Map Entry Schema

```go
UnderlierXxx: {
    name:           "XxxMessage",       // PascalCase; lingo derives the struct name and kebab ID from this
    msg:            "The value is {{.Value}}.", // the Other string; use plain text for static messages
    underlyingType: UnderlierXxx,       // must match the map key
    category:       categoryGeneral,    // categoryGeneral | categoryError | categoryCobra
    isError:        false,              // true for all error UnderlyingTypes
    fields: []field{                    // one entry per {{.Token}} in msg; empty for static messages
        {name: "Value", tp: "string"},
    },
    wrap: false,                        // true only for wrapping error types
},
```

### Field rules

| Field           | Required | Notes                                                                 |
|-----------------|----------|-----------------------------------------------------------------------|
| `name`          | always   | PascalCase; lingo strips `TemplData` and converts to kebab for the ID |
| `msg`           | always   | Plain text for static; `{{.FieldName}}` tokens for dynamic            |
| `underlyingType`| always   | Must match the map key constant                                        |
| `category`      | always   | `categoryGeneral`, `categoryError`, or `categoryCobra`                |
| `isError`       | always   | `false` for Cobra and General types; `true` for all Error types        |
| `fields`        | dynamic only | One `{name, tp}` per `{{.Token}}` in `msg`; must be empty for static types |
| `wrap`          | wrapping types | `true` for `StaticErrorWrapper`, `StaticErrorWrapperMsg`, `DynamicErrorWrapper` |

---

## 4. Choosing the Right `UnderlyingType`

Use the decision tree below to classify a new message, then confirm the
constraints in section 5.

```
Is it a Cobra command/flag description?
├── Yes → Is msg fixed (no {{.Field}} tokens)?
│         ├── Yes → UnderlyingTypeStaticCobra
│         └── No  → UnderlyingTypeDynamicCobra
└── No  → Is it an error?
          ├── No  → Is msg fixed?
          │         ├── Yes → UnderlyingTypeStaticGeneral
          │         └── No  → UnderlyingTypeDynamicGeneral
          └── Yes → Does it wrap another error?
                    ├── No  → Is msg fixed?
                    │         ├── Yes, consumed directly → UnderlyingTypeStaticError
                    │         ├── Yes, wrapped by others → UnderlyingTypeSentinelError
                    │         └── No                     → UnderlyingTypeDynamicError
                    └── Yes → Is msg fixed?
                              ├── Yes, wrapped text NOT in msg → UnderlyingTypeStaticErrorWrapper
                              ├── Yes, wrapped text IN msg     → UnderlyingTypeStaticErrorWrapperMsg
                              └── No                           → UnderlyingTypeDynamicErrorWrapper
```

---

## 5. Per-Type Constraints

| UnderlyingType                  | `isError` | `fields`       | `wrap`  | Notes                                                              |
|---------------------------------|-----------|----------------|---------|--------------------------------------------------------------------|
| `StaticCobra`                   | false     | empty          | false   |                                                                    |
| `DynamicCobra`                  | false     | non-empty      | false   | Each `{{.Token}}` in `msg` needs a matching `fields` entry         |
| `StaticGeneral`                 | false     | empty          | false   |                                                                    |
| `DynamicGeneral`                | false     | non-empty      | false   | Each `{{.Token}}` in `msg` needs a matching `fields` entry         |
| `StaticError`                   | true      | empty          | false   | Standalone error; consumed directly via `ErrXxx`                   |
| `SentinelError`                 | true      | empty          | false   | Core error designed to be wrapped by an outer error via `errors.Is`|
| `StaticErrorWrapper`            | true      | empty          | true    | Fixed msg; wrapped error present in chain but not in translation   |
| `StaticErrorWrapperMsg`         | true      | empty          | true    | Fixed msg; use `{{.Wrapped}}` in `msg` to include wrapped text     |
| `DynamicError`                  | true      | non-empty      | false   | `Wrapped` must not appear in `fields`                              |
| `DynamicErrorWrapper`           | true      | non-empty      | true    | `Wrapped` must not appear in `fields`; lingo adds it automatically |

---

## 6. Message ID Convention

Lingo derives the message ID automatically from `name` and `underlyingType`.
The format is:

```
<kebab-slug>.<type-suffix>
```

Where `<kebab-slug>` is `name` converted from `PascalCase` to `kebab-case`,
and `<type-suffix>` is one of:

| UnderlyingType family           | Suffix            |
|---------------------------------|-------------------|
| Cobra / General (any)           | *(none)*          |
| StaticError / ErrorWrapper types| `.static-error`   |
| SentinelError                   | `.sentinel-error` |
| DynamicError / DynamicErrorWrapper | `.dynamic-error` |

There is no project name in the ID. Lingo enforces this automatically.

---

## 7. Common Mistakes to Avoid

| Mistake | Correct behaviour |
|---|---|
| Editing a `*-auto.go` file directly | Never. Edit `underlying-templ-data.go` and run lingo |
| Adding a canonical `UnderlyingType` constant without syncing `cmd/lingo/main.go` | Follow the SYNC NOTICE checklist in `main.go` first |
| Using `StaticError` for a core error intended to be wrapped by outer errors | Use `SentinelError` - intent and ID suffix differ |
| Using `StaticErrorWrapper` when the wrapped error's text must appear in the translation | Use `StaticErrorWrapperMsg` and include `{{.Wrapped}}` in `msg` |
| Including a `Wrapped` entry in `fields` for `DynamicError` or `DynamicErrorWrapper` | Lingo adds `Wrapped` automatically for wrapping types; never put it in `fields` |
| Leaving `fields` empty for a dynamic type | Every `{{.Token}}` in `msg` must have a matching `fields` entry |
| Including a project name in the message ID | IDs are `<kebab-slug>.<type-suffix>` only - lingo enforces this |
| Running lingo before syncing the enum duplicate in `cmd/lingo/main.go` | Sync first, then run lingo |
